# 📋 Root 治理層驗證清單

## 🎯 目的
本清單用於**手動驗證** AI 代理人或協作者的交付是否符合治理要求。  
適用於：行動裝置快速驗證、自動化 gate 失效時的備援驗證。

---

## ✅ 快速驗證流程（3 分鐘完成）

### 第 1 步：檢查四大核心證據（30 秒）
在 PR 描述中確認以下四項：

- [ ] **repo**: 連結格式正確且指向本專案
  - 格式：`https://github.com/MachineNativeOps/MachineNativeOps`
  
- [ ] **branch**: 分支名稱有意義（非 tmp/test/new）
  - 格式：`feature/<name>` 或 `fix/<name>`
  
- [ ] **commit**: 完整 40 字元 SHA
  - 格式：40 個小寫英數字元
  - 驗證：點擊 PR 的 Commits 標籤，對比 SHA 是否一致
  
- [ ] **PR**: 本 PR 的 URL
  - 格式：`https://github.com/MachineNativeOps/MachineNativeOps/pull/[NUMBER]`

### 第 2 步：檢查 Files Changed（1 分鐘）
點擊 PR 的 **Files changed** 標籤：

- [ ] 確認有實際的檔案變更（不是空 PR）
- [ ] 變更內容與 PR 描述一致
- [ ] 沒有未授權的檔案修改（特別是 `root/` 和 `.github/`）

### 第 3 步：檢查 Checks 狀態（30 秒）
在 PR 頁面查看 GitHub Actions 狀態：

- [ ] ✅ `gate-pr-evidence` 通過（四證據驗證）
- [ ] ✅ `gate-root-naming` 通過（若有變更 root/）
- [ ] ✅ 其他必要的 CI 檢查通過

### 第 4 步：治理層變更特別檢查（1 分鐘）
若 PR 變更了 `root/` 或 `.github/`：

- [ ] PR 描述中有明確說明變更理由
- [ ] 提供了影響評估（會破壞什麼）
- [ ] 提供了回滾方式（如何撤回）
- [ ] 第五證據已勾選（命名規範遵循）

---

## 🚫 立即退回的情況（不需討論）

以下任一情況出現，**直接關閉 PR 並要求重新提交**：

### 證據缺失
- [ ] 四大證據任一項缺失或格式錯誤
- [ ] commit SHA 不是 40 字元
- [ ] PR URL 指向錯誤的 repo 或 fork

### 內容問題
- [ ] Files changed 是空的（沒有實際變更）
- [ ] 變更內容與描述完全不符
- [ ] 包含未授權的大規模重構

### 治理違規
- [ ] 直接 push 到 main（繞過 PR）
- [ ] 變更 `root/` 但未說明理由/影響/回滾
- [ ] 命名規範檢查失敗（紅叉）

## 📱 行動裝置驗證技巧

### 快速複製 SHA 驗證
1. 長按 PR 描述中的 commit SHA
2. 選擇「複製」
3. 點擊 PR 的 Commits 標籤
4. 對比第一個 commit 的 SHA 是否一致

### 快速查看 Files Changed
1. 在 PR 頁面向下滑動
2. 點擊「Files changed」標籤
3. 快速瀏覽變更的檔案路徑
4. 確認沒有意外的檔案被修改

### 快速檢查 Checks 狀態
1. 在 PR 頁面頂部查看狀態圖示
2. ✅ 綠勾 = 通過
3. ❌ 紅叉 = 失敗（立即退回）
4. 🟡 黃圈 = 執行中（等待完成）

## 🔍 深度驗證（可選，5-10 分鐘）

### 程式碼品質檢查
- [ ] 程式碼風格符合專案規範
- [ ] 沒有明顯的安全漏洞
- [ ] 測試覆蓋率足夠

### 架構一致性檢查
- [ ] 變更符合專案架構設計
- [ ] 沒有引入循環依賴
- [ ] 模組邊界清晰

### 文檔完整性檢查
- [ ] 相關文檔已更新
- [ ] 變更日誌已記錄
- [ ] API 文檔已同步

## 🆘 常見問題

### Q: 如果 GitHub Actions 失效怎麼辦？
A: 使用本清單進行手動驗證，特別注意第 1、2、4 步。

### Q: 如果代理人堅稱「已經做了」但找不到證據？
A: 直接退回，要求提供四大證據。不追查、不討論。

### Q: 如果 PR 很大，驗證會很久嗎？
A: 快速驗證只需 3 分鐘。大 PR 應該要求代理人拆分。

### Q: 如果在手機上看不清楚程式碼？
A: 只驗證第 1-4 步即可。程式碼品質可以在電腦上再審。
