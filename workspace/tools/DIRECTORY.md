# tools

## 目錄職責
此目錄包含 MachineNativeOps Enterprise 的開發、部署和維護工具集合。它提供各種自動化腳本、代碼生成工具、質量檢查工具和實用程序，支持整個軟體開發生命週期。此目錄與 `scripts/` 目錄協作，`scripts/` 專注於構建和部署流程，而 `tools/` 專注於開發輔助和代碼質量工具，共同為開發團隊提供完整的工具鏈支持。

## 職責分離說明

此目錄實現了工具功能的職責分離：

1. **AI 工具層**：`ai/` 專注於 AI 輔助開發，不包含傳統工具邏輯
2. **自動化工具層**：`automation/` 專注於流程自動化，不涉及具體業務邏輯
3. **質量檢查層**：`governance/` 和 `verify_*.py` 專注於質量保證，獨立於功能開發
4. **代碼生成層**：`generators/` 專注於代碼生成，不包含運行時邏輯
5. **重構工具層**：`refactor/` 和 `validate-*.py` 專注於代碼重構，不影響業務功能
6. **文檔工具層**：`docs/` 專注於文檔處理，與代碼開發分離
7. **系統工具層**：`path_tools/`, `utilities/` 專注於系統級工具，獨立於應用邏輯

每個工具類別都有明確的職責邊界，確保工具的專業性和可維護性。

## 設計原則

### 單一職責原則 (SRP) 遵循

1. **工具職責單一化**：
   - 每個工具專注於特定的開發或維護任務
   - 避免在一個工具中混合多種不同類型的功能
   - 確保工具的專注性和易用性

2. **腳本職責專一化**：
   - 每個腳本文件專注於特定的工具功能
   - 驗證腳本只負責檢查，不包含修復邏輯
   - 生成腳本只負責創建，不包含管理邏輯

3. **目錄職責專一化**：
   - 每個子目錄專注於特定類型的工具集合
   - 實驗性工具放在 `_scratch/` 中，不影響穩定工具
   - 按工具功能進行分類，避免混合存放

### 未來維護注意事項

1. **添加新工具時**：
   - 確定工具屬於哪個現有分類或需要創建新分類
   - 保持工具的專注性，不要創建多功能的大雜燴工具
   - 確保新工具與現有工具鏈的兼容性

2. **修改現有工具時**：
   - 維持工具的核心職責不變
   - 不要在工具中添加與其主要功能無關的特性
   - 確保修改不會破壞工具的穩定性

3. **工具演化策略**：
   - 優先通過配置和參數進行功能擴展
   - 實驗性功能先在 `_scratch/` 中驗證
   - 保持工具的簡潔性和專業性
   - 定期更新工具依賴和適配新的開發環境
